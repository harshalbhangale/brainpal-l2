# BrainPal landing page sections and page assembly (v1)

This change builds out the visible landing page for BrainPal: seven section components (navbar, hero, features, testimonials, pricing, CTA, footer) composed into a single `app/page.tsx`. All motion is driven by CSS keyframes already defined in the Tailwind config and `globals.css` from a prior commit; the new code applies those animations via utility classes with staggered inline `animationDelay` values. The project targets Next.js 14 App Router with TypeScript and Tailwind, using hand-rolled shadcn-style UI primitives (Button, Card, Badge, Input) that already exist on `main`.

**Watch for:** All animations fire immediately on page load rather than on scroll into view (confirmed). The CTA email form has no submit handler or validation (confirmed). No motion library is integrated despite the user requesting one (confirmed).

## High-level view

The page is a vertical stack of server and client components. Three components (`Navbar`, `Hero`, `CTA`) are marked `"use client"` for interactivity; the rest are server components. The animation strategy uses `opacity-0` as the initial state with `[animation-fill-mode:forwards]` to hold the end state, creating reveal effects. Because no scroll-intersection observer is employed, every animation triggers the moment the component mounts — content below the fold animates before the user scrolls to it, so staggered delays become invisible on long pages.

No motion library (Framer Motion, GSAP, etc.) is integrated despite the user request mentioning "some great motion design library." The CSS-only approach handles entrance animations adequately but cannot support scroll-triggered reveals, spring physics, gesture interactions, or orchestrated sequences that a library would provide.

The CTA section renders an email input and button with no form submission logic — it's a visual placeholder with no handler, no validation, and no feedback on interaction.

## Animation strategy fires on mount, not on scroll

Every section heading, card, and CTA element starts at `opacity-0` with a CSS animation class and `[animation-fill-mode:forwards]`. The animations play immediately when the component mounts. For below-the-fold sections (features, testimonials, pricing, CTA), the animations complete before the user scrolls there. This defeats the purpose of staggered card reveals in the features grid (delays of 100ms–600ms across 6 cards) and the pricing tier entrance (150ms–450ms).

The fix is wrapping animated elements in an Intersection Observer that adds the animation class only when the element enters the viewport:

```tsx
function useInView(ref: React.RefObject<HTMLElement>) {
  const [inView, setInView] = useState(false);
  useEffect(() => {
    const obs = new IntersectionObserver(([e]) => {
      if (e.isIntersecting) { setInView(true); obs.disconnect(); }
    }, { threshold: 0.1 });
    if (ref.current) obs.observe(ref.current);
    return () => obs.disconnect();
  }, []);
  return inView;
}
```

Adopting Framer Motion would solve this and also fulfill the user's explicit request for a motion design library — unlocking scroll-linked animations, spring-based interactions, and layout transitions that pure CSS can't express.

## CTA section is visual-only

The CTA component renders an `<Input type="email">` and a `<Button>` wrapped in a plain `<div>`, not a `<form>`. There's no `onSubmit`, no state variable capturing the email value, no validation feedback. Clicking "Get Early Access" does nothing. It should either be wrapped in a `<form>` with an action (even `#`) or have an explicit comment marking it as a placeholder for future integration.

## Navbar mobile menu accessibility

The mobile menu hamburger button lacks `aria-expanded={mobileOpen}`. The mobile menu container appears conditionally but has no ARIA landmark or label. A screen reader user tapping the button gets no feedback about whether the menu opened. Adding `aria-expanded` to the button and wrapping the menu links in a `<nav aria-label="Mobile navigation">` would close this gap.

The scroll listener should also pass `{ passive: true }` to signal the browser it won't call `preventDefault`, avoiding potential scroll-jank on mobile devices.

## Duplicate smooth-scroll declarations

`globals.css` has `html { scroll-behavior: smooth; }` and `app/page.tsx` adds `className="scroll-smooth"` to `<main>`. The CSS rule on `html` already applies smooth scrolling globally; the Tailwind utility on `<main>` is redundant (`scroll-behavior` on a non-scrolling container has no effect unless it has `overflow: auto/scroll`). Remove the `scroll-smooth` class from `<main>`.
