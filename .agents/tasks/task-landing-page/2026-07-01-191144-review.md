# BrainPal landing page with scroll-triggered animations (v2)

This change delivers the full BrainPal landing page: seven section components composed into `app/page.tsx`, a custom `useInView` hook for scroll-triggered animation via Intersection Observer, a functional CTA email form with validation and success state, and accessibility improvements to the navbar mobile menu. The CSS-only animation approach (keyframes defined on `main`) is now gated behind viewport intersection so below-the-fold content animates on scroll. No motion library is installed due to the INTEGRATIONS_ONLY network constraint; a code comment notes framer-motion as a future enhancement.

**Watch for:** The `useInView` hook has a referential-stability issue with its `options` parameter that can cause infinite re-renders if a caller ever passes an inline object (likely). Hero animations still fire on mount without scroll gating, which is intentional for above-the-fold content but worth confirming as a deliberate choice (possible).

## High-level view

All five issues from the v1 review have been addressed. Scroll-triggered animation now works through a shared `useInView` hook applied to Features, Testimonials, Pricing, and CTA. The hook is well-structured — one-shot observer that disconnects after first intersection — but has a dependency-array bug that doesn't bite today only because no caller passes options.

Hero is the one section that still animates on mount while the other four use `useInView`. This is fine for above-the-fold content but the inconsistency is undocumented, leaving it vulnerable to someone "fixing" it later and causing a flash of invisible content.

## useInView dependency array creates a latent re-render trap

The hook's `useEffect` lists `options` in its dependency array:

```ts
useEffect(() => {
  // ...observer setup...
}, [options]);
```

Today every caller invokes `useInView<HTMLElement>()` with no arguments, so `options` is `undefined` on every render and the effect runs once. If any future caller passes `{ threshold: 0.2 }` inline — the obvious usage pattern — React will see a new object reference on every render, tear down the observer, and re-create it in a loop until intersection fires. The one-shot `disconnect()` inside the callback limits the blast radius (the loop stops once the element is in view), but the repeated observer churn during initial scroll is wasteful and may cause animation flicker.

The fix is trivial: since the hook is one-shot (it disconnects after first intersection and never re-observes), removing `options` from the dependency array and reading it via a ref is safe. Alternatively, an `eslint-disable` with a comment explaining that options are only read on mount works too.

## CTA form lacks reset path for backend failures

There's no way to reset `submitted` back to false. If this form ever gains a real backend that can fail, the current two-state boolean (`submitted` true/false) would need to become a loading/error/success state machine. Worth noting as a future-proofing gap even though the client-only demo doesn't hit this path today.
